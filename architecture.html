GitHub Actions — Automation
Scheduled jobs + one-time migrations

YAML workflows that run in the cloud. Used for data migration (Jekyll → Supabase), scheduled article generation, and newsletter sending. Key advantage: Actions run from GitHub's IP — bypasses Supabase IP allowlists that block this server.

  • migrate.yml — one-time: reads _posts/*.md, upserts to Supabase
  • aggregate.yml — hourly: RSS fetch + AI rewrite → drafts table
  • monthly-reports.yml — 1st of month: generate summary article
  • newsletter.yml — Fridays: send digest via Buttondown API
  • Receives OPENAI_API_KEY, SUPABASE_SERVICE_ROLE_KEY as secrets

Vercel — Hosting + Edge
Builds and serves the Next.js app globally

Listens to GitHub pushes via webhook. On every push to main, Vercel pulls the code, runs npm run build, and deploys globally across its edge network in ~30 seconds.

  • Framework: Next.js 15 App Router (server + client components)
  • Build: npm run build → .next/ output
  • Preview deploys: every PR gets its own URL (*.vercel.app)
  • Production: custom domain (locreport.com)
  • Env vars: set in Vercel Dashboard → injected at build + runtime
  • Edge functions: API routes run serverless (e.g. /api/ingest)
  • ISR / revalidate: pages cache and refresh on schedule
  • vercel.json controls build, redirects (.html → clean URLs)
GitHub Webhook Vercel API Edge Network (CDN)
SB
Supabase — Database + Auth
PostgreSQL + REST API + Auth + RLS

Managed PostgreSQL with a built-in REST API (PostgREST). Every article, draft, and RSS source is stored here. Two access tiers: public reads via anon key, admin writes via service role key.

  • Tables: articles, drafts, rss_sources
  • RLS: Row Level Security — anon can SELECT articles; writes require service key
  • Anon key (sb_publishable_*) — safe to expose in browser, read-only
  • Service role key (sb_secret_*) — bypasses RLS, never in browser
  • Auth: email+password sessions for admin panel
  • REST API: https://[project].supabase.co/rest/v1/articles
  • Schema migrations: run SQL in Studio → SQL Editor
  • Network restrictions: set in Supabase → Project Settings

🔗 API & Connection Map — who calls what

Claude → GitHub
GitHub MCP server — Claude reads/creates PRs, pushes files, checks CI status. Uses a GitHub Personal Access Token stored in the Claude Code session.
GitHub → Vercel
GitHub App webhook — Vercel listens for push events on connected repos. Fires a build automatically on every push to main.
Vercel → Supabase
PostgREST REST API over HTTPS — Next.js server components call supabase.from('articles').select() using the anon key for public reads.
Vercel API → Supabase
Service role key — API routes (/api/articles, /api/drafts) use the service key server-side for writes. Never exposed to the browser.
Vercel → OpenAI
OpenAI Chat Completions API — called from /api/ingest and /api/compose server routes only. Key injected as OPENAI_API_KEY env var.
GitHub Actions → Supabase
Service role key + PostgREST — migration scripts, aggregate jobs, and monthly report generation call Supabase directly from GitHub Actions runners.
GitHub Actions → OpenAI
OpenAI API — scheduled aggregate.yml workflow fetches RSS, sends to OpenAI, saves drafts to Supabase. Runs every hour at :33.
DNS → Vercel
CNAME / A record at domain registrar points locreport.com to Vercel's edge. Vercel auto-issues TLS cert. Old GitHub Pages becomes inactive.
Browser → Vercel
HTTPS — visitor requests hit Vercel's CDN edge first. Static pages served from cache; dynamic routes invoke serverless functions that query Supabase.
Browser → Supabase Auth
Supabase Auth SDK — login page calls supabase.auth.signInWithPassword() in the browser. Session cookie stored; middleware validates it on every /admin/* request.

🔑 Environment Variables — what goes where

Set in: Vercel env vars + GitHub Actions secrets
Value: https://[project-id].supabase.co
Visible in browser: ✅ safe (public)
NEXT_PUBLIC_SUPABASE_ANON_KEY
Set in: Vercel env vars
Value: sb_publishable_… (new format) or eyJ… (legacy)
Visible in browser: ✅ safe (read-only by RLS)
SUPABASE_SERVICE_ROLE_KEY
Set in: Vercel env vars (server-only) + GitHub Actions secrets
Value: sb_secret_…
Visible in browser: ❌ never — bypasses RLS entirely
OPENAI_API_KEY
Set in: Vercel env vars + GitHub Actions secrets
Used by: /api/ingest, /api/compose, aggregate workflow
Visible in browser: ❌ never

Set in: Vercel env vars
Value: parasion@proton.me

📋 The Recipe — how to build this from scratch with Claude Code

1
Create Supabase project
Go to supabase.com → New project. Copy the project URL and both keys (anon + service role). Run your supabase-schema.sql in SQL Editor → Tables are ready.
2
Create Vercel project
Go to vercel.com → Import Git Repository → pick your GitHub repo. Set framework: Next.js. Add all env vars (Supabase URL, keys, OpenAI key). Vercel creates a webhook on your GitHub repo automatically.
3
Add GitHub Actions secrets
GitHub repo → Settings → Secrets and variables → Actions. Add NEXT_PUBLIC_SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, OPENAI_API_KEY. These power the migration and automation workflows.
4
Run the data migration
GitHub → Actions → "Migrate Jekyll posts to Supabase" → Run workflow. It reads all _posts/*.md files, parses YAML front matter, and upserts to the articles table. ~909 articles in ~2 minutes.
5
Tell Claude Code to build features
Open a Claude Code session at claude.ai/code. Describe features in plain English. Claude reads your codebase, edits files, and pushes to a feature branch. You review the PR, merge → Vercel auto-deploys in ~30 seconds.
e.g.: "add an inline search bar to the nav" "fix the footer layout on mobile"
6
Point your domain to Vercel
Vercel Dashboard → Project → Settings → Domains → Add domain. Copy the CNAME value Vercel gives you. Go to your domain registrar → DNS → add CNAME record. Wait 5–30 min for propagation. Vercel issues TLS automatically.
7
Enable automation workflows
GitHub Actions workflows run on schedule: RSS ingest every hour, monthly reports on 1st of month, newsletter on Fridays. All AI content flows: RSS feed → GitHub Action → OpenAI → Supabase drafts → admin review → published article.
8
Ongoing: use Claude Code to evolve the product
Any future feature — new page, layout change, new API integration, bug fix — is a conversation with Claude Code. No deployment knowledge required. Push → PR → merge → live. Claude Code can also watch PRs and auto-fix CI failures.
